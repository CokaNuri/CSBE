# 소프트웨어 고장 사례

> 핵심 : 소프트웨어 결함은 돈·생명까지 앗아간다 <br>
> → 그래서 소프트웨어 공학이 필요하다 각 사례를 "어떤 품질이 무너졌나"로 읽어보자

## 화성 기후 관측 위성 (1999)

- 원인: **단위 불일치**. 국제 표준 단위인 1(N·s) 로 표시해야 하는데 미국식인 4.45(lb·s)로 표시
- 결과: MCO 위성이 사라짐 -> 큰 경제적 손실
- 교훈: 표준을 잘 따르자. 문서화를 잘 하자.

## 자동차 해킹

- 원격으로 차량 제어 시스템에 침입해 핸들·브레이크·엔진 조작 가능 (예: 2015 짚 체로키)
- 교훈: **보안성**이 곧 **안전성**과 직결. 차가 인터넷에 연결되며 보안 결함이 인명 위협으로

## 보잉 737 MAX8 추락

- 원인: **MCAS**라는 SW기반 제어 시스템에 입력되는 노즈 센서 값 오류<br>
  하나의 노즈 센서 값만 이용하여 동작하도록 개발
- 결과: 두 사고로 346명 사망, 전 세계 737 MAX 약 20개월 운항 정지.
- 교훈: **단일 장애점**. 최소 두 센서를 교차 확인했어야 함. 견고성·안전성 설계 실패

## 테슬라 자율주행

- 오토파일럿/FSD 사고. 운전자 과신 + AI가 특정 상황 오인
- 결과 : 테슬라 모델 S 오토파일럿 때문에 운전자 사망 사고
- 교훈: AI 견고성, 보조 시스템을 최종 결정에 쓰면 안 된다는 범위 준수

## 기타 소프트웨어 사고 일지

- Therac-25(방사선 치료기 SW 결함 → 환자 사망)
- Ariane 5(숫자 변환 오버플로로 로켓 폭발)
- 산업, 과학기술 분야에서 SW 결함으로 인한 사고 발생

## 소프트웨어 개발이 어려운 이유

> SW 개발은 창의적인 활동 -> 엔지니어의 지식과 형험에 따라 결과물 천차만별

## 소프트웨어 특성

> 왜 소프트웨어를 단순한 개발 활동으로 간주할 수 없는가?

> 왜 체계적인 공학적인 기술을 도입해야 하는가?

- 개발 비용 : 분석 및 설계, 테스트 과정이 코드 개발보다 더 중요하다
  - 분석 및 설계(40%), 테스트(40%), 코드 개발(20%)

- 유지보수 : 운영 유지 비용이 개발 비용의 2배

- 고장 그래프 : SW는 일정 시간 후 변경 과정을 거친다
  - 수정이 반복되면, 코드는 더 복잡해지고 이해하기 어려워져 오류 발생 ⬆️

## 인공지능 시대의 소프트웨어 위기

- 너무 많은 학습 알고리즘이 무분별하게 응용 SW 개발에 적용되는 문제
  - 자율 주행차, 스마트 팩토리 로봇 제어 등과 같은 안전 중심 SW에도 적용 ➡️ 인명 피해
- 훈련된 데이터를 조금만 변형해도 다른 데이터로 인식하는 문제

# 소프트웨어 공학 기술의 적용

> 위기의 답으로 등장한 것이 소프트웨어 공학

## 소프트웨어 공학적 기법의 종류

> 소프트웨어 품질을 높이고, 신속 정확하게 SW 전달하는게 목표

- **구조적 프로그래밍**: 순차·선택·반복 세 구조로 프로그램 구성
- **객체지향 프로그래밍**: 소프트웨어 직관적 이해, 유지보수성 및 재사용성 ⬆️
- **소프트웨어 컴포넌트 및 재사용**: 검증된 부품을 조립해 개발 시간 단축, 대규모 SW 개발
- **개발 프로세스**: 폭포수·애자일 등 체계적 SW 개발 절차
- **검사 및 검증**: 테스트 등을 통해 제대로 SW가 동작하는지 확인
- **형상 관리**: 버전·변경 이력 통제
- **소프트웨어 아키텍처**: 변경에 강한 SW를 위해 상호 독립성이 높은 구성 요소를 식별하고 관계를 정의

## 소프트웨어 공학의 정의와 원리

### 정의

- SW 위기에서 벗어나기 위한 공학적 기법
- 전체 수명주기 지원
- 사용자 요구사항 충족
- 좋은 품질의 소프트웨어 개발

### 원리

| 원리          | 설명                                            |
| ------------- | ----------------------------------------------- |
| 엄격성·정형성 | 지나친 자유스러움 제한                          |
| 관심사 분리   | 복잡한 문제를 분리하여 별도로 관리              |
| 모듈화        | 독립된 작은 단위로 만들어 쉽게 분리·조합 👌     |
| 추상화        | 핵심만 남기고 세부사항은 감춤 ➡️ 관심사 잘 표현 |
| 변경 예측     | 미래의 변경을 미리 고려해 설계                  |
| 일반화        | 더 넓게 재사용 가능한 형태로                    |
| 점진성        | 조금씩 단계적으로 발전시킴                      |
| 명세화        | 무엇을 만들지 명확히 정의·문서화                |

최종 정리

소프트웨어 제대로 안 만들면 재산,인명 사고 발생한다
-> 좋은 품질의 소프트웨어 만들어야 한다
-> SW의 어떤 특성 때문에 좋은 소프트웨어 개발이 어려운가?
-> 1.전체 수명을 다 관리해야 한다
-> 단순 개발보다는 설계,테스트,유지보수에 많은 비용이 든다
-> 관심사의 분리, 모듈화, 추상화
-> 2.다양한 사람들의 요구사항 다 충족시켜야 한다
-> 의사소통 비용 많다
-> 명세화, 문서화, 표준화 잘 해야 한다
