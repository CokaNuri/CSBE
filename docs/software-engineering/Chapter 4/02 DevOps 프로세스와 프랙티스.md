# DevOps 프로세스와 프랙티스 

## DevOps 성공 전략 
> 사일로 개발 방식을 타파하고 부서 간 효과적 의사소통이 필요 

### 성공적인 DevOps 구현을 위한 권장 사항

- DevOps 필요성 평가
- 기존 사일로 방식 타파
- 고객 중심 만족도 향상
- 작은 시작과 점진적 확장
- 가능한 모든 곳의 자동화
- 호환성 보장 도구
- 개인화 함꼐 팀 성과 검토
- 프로젝트 정보의 실시간 가시성 확보
- 지속적 통합 및 배포
- 모니터링 및 피드백을 통한 개선

## DevOps 프로세스 

> 애플리케이션 배포 및 운영 환경과 개발 및 테스트 환경 간 격차 해소

- 계획 
  - 요구 사항 명세
  - 배포 척도 조정
  - 배포 게획, 시기, 비즈니스 사례 정의

- 코딩
  - 소프트웨어 설계
  - 소프트웨어 제품 형상 설계·정의
  - 소스 코드 작성
  - 단위 테스트 수행

- 빌드
  - 통합 테스트 수행
  - 제품에 대한 형상 구성
  - 배포를 위해 개발 애플리케이션 준비
  - 애플리케이션 버전 관리

- 테스트
  - 요구사항 만족도 테스트
  - 회귀 테스트
  - 보안 및 취약성 분석
  - 성능 테스트
  - 소프트웨어 형상 테스트

- 릴리즈
  - 형상에 따라 패키징
  - 애플리케이션 배포
  
- 설치
  - 배포된 애플리케이션을 사용자 환경에 설치
  - 스토리지, 데이터베이스, 네트워크 등의 인프라 구축
  - 설치된 애플리케이션을 사용자에게 오픈
- 운영 및 모니터링
  - 인프라 구성 요소의 성능 모니터링
  - 사용자의 반응 및 경험을 모니터링
  - 개발 척도의 달성 여부 확인
  - 개발 팀에 모니터링 결과 제공

제품 형상을 설계, 형상 구성, 형상 테스트 이게 무엇인가?
패키징은 정확히 무엇인가? 

## DevOps와 클라우드 

> 클라우드를 통해 DevOps를 구현하면 편리하게 서비스 개발 및 배포가 가능

- On-Demand 시점의 가용성 및 접근성 제공받을 수 있음
- 다양한 테스트가 쉽게 이루어질 수 잇음

### 풀 스택 배포 

> 클라우드 환경에 애플리케이션 서비스 배포와 실행 환경 구성에 대한 배포를 함께 진행 

- 분리 배포 : 클라우드 환경 프로비저닝을 애플리케이션 서비스 배포와 분리하여 배포 

클라우드 환경 프로비저닝이 무엇인가? 실행 환경 구성이 무엇인가?

### 클라우드 서비스 모델 

- IaaS : 네트워크,저자소,서버 등을 포함하는 가상화된 인프라만 제공, 개발과 인프라 관리간 사일로 존재
- Paas : 애플리케이션과 데이터 생성 및 관리만 고민하면 됨

## DevOps 측정 척도

> 신속하고 지속적으로 배포하는 게 목적 

- 설치 주기 
- 설치 규모
- 설치 시간
- **소요 시간** : 작업을 시작하여 배포될 때까지 소요되는 시간
- 고객 티켓
- 자동 테스트 성공률
- 결함 제거율
- 가용성
- 서비스 수준 협약
- 서비스 활용 및 트래픽
- 서비스 성능
- 결함 발견 간 평균 시간
- 결함 복구 간 평균 시간

이렇게 많은 측정 척도 중 가장 중요한 건 무엇일까?
