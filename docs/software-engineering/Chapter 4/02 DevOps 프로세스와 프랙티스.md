# 전통적인 소프트웨어 개발 프로세스

## 소프트웨어 프로세스란?

> 서로 관련 있는 활동을 단계로 분리하여 구분한 것

> 구축, 배포, 설치, 운영 시 수행해야 하는 절차

## 소프트웨어 프로세스의 필요성

| 구분 | 블랙박스 프로세스 | 투명한 프로세스 |
|------|------------------|-----------------|
| 구조 | 하나의 통합된 개발 과정 | 여러 단계로 분리 |
| 사용자 확인 | 어려움 | 각 단계마다 요구사항 확인 가능 |
| 요구사항 만족 | 만족 못 할 확률 ⬆ | 사용자 만족도 ⬆ |
| 결과 | 과도한 수정·재개발 발생 | 수정·재개발 비용 ⬇, 출시 기간 ⬇, 최종 품질 ⬆ |

## 전통적인 프로세스 모델

### 폭포수 모델

> 순차적인 단계로 소프트웨어를 개발하고, 각 단계마다 정해진 문서를 작성하는 모델

- 단계 흐름: **요구사항 분석 → 설계 → 개발 → 테스트 → 배포**
- 단계가 명확해 관리가 단순함
- 개발 과정에서 요구되는 변경을 수용하기 어려움

### 프로토타입 모델

> 기능을 모두 갖춘 SW가 아니라, 인터페이스만 보여주거나 핵심 모듈만 우선 개발해 사용자에게 제공하는 모델

- 1차: 요구사항 검증을 위한 프로토타입 개발 → 2차: 폭포수 모델에 따라 전체 시스템 개발
- 요구사항을 프로토타입으로 빠르게 검증해 비용·시간 단축
- 사용자와 개발자 간 의사소통이 원활해짐
- 빠른 오류 탐지 가능
- 단점: 사용자 검증 이후에도 변경이 발생할 수 있는데, 이 점은 여전히 충분히 고려하지 못함

### 나선형 모델

> 위험을 **최소화**하기 위해 점진적으로 전체 시스템을 개발해 나가는 모델

> 핵심 키워드: **위험 최소화 + 점진적 개발**

사이클 하나당 4개의 활동을 반복한다.

| 활동 | 내용 |
|------|------|
| 목표 설정 | 목표 정의, 대안 분석, 제한 사항 식별 |
| 위험 분석 | 예측되는 위험 추출, 대처 방안 수립 |
| 개발 및 검증 | SW 구현, 테스트 |
| 평가 및 다음 계획 수립 | 산출물 평가, 다음 사이클 계획 |

| 장점 | 단점 |
|------|------|
| 위험성이 큰 프로젝트 관리에 적합 | 프로젝트 기간이 길다 |
| 요구사항을 상세히 적용 가능 | 사이클이 많아질수록 관리가 어렵다 |
| 변경되는 요구사항 반영이 쉬움 | 위험 분석에 비용과 전문 지식이 필요 |
| 사용자 만족도와 품질이 높아짐 | |

<details><summary>나선형 모델은 어떤 분야에서 주로 사용되는가?</summary>

위험이 크고 규모가 크며 요구사항이 불확실한 프로젝트에 주로 쓰인다. <br>
대표적으로 국방·군사 시스템, 항공우주, 대규모 기업용 시스템, 장기 연구개발 성격의 프로젝트  <br>
공통점은 실패했을 때의 비용이 매우 크고, 그래서 위험 관리 자체가 개발만큼 중요한 영역
</details>

<details><summary>위험이란 어떤 것인가?</summary>

불명확하거나 자주 바뀌는 요구사항, 검증되지 않은 새 기술의 사용, 일정 지연, 예산 초과, 인력 부족, 성능 미달 등  

</details>

### V 모델

> 폭포수 모델의 확장판으로, 각 개발 단계마다 대응하는 테스트 단계가 존재하는 모델

| 개발(검증 명세) 단계 | 대응하는 테스트 단계 |
|----------------------|----------------------|
| 요구사항 분석 | 인수 테스트 |
| 시스템 설계 | 시스템 테스트 |
| 상세(아키텍처) 설계 | 통합 테스트 |
| 모듈(단위) 설계 | 단위 테스트 |
| 구현(코딩) | |

- 단계적인 검증·확인 과정을 거쳐 오류를 줄일 수 있다
- 요구사항이 정확히 이해되는 작은 시스템 개발 시 매우 유용
- 단점: 변경을 다루기가 쉽지 않다

<details><summary>V 모델은 정확히 어떤 순서로 개발이 진행되는가?</summary>

V자 모양으로, 왼쪽을 따라 위에서 아래로 설계를 구체화하며 내려가고 → 바닥에서 구현(코딩)을 한 뒤 → 오른쪽을 따라 아래에서 위로 테스트하며 올라온다.

순서: 요구사항 분석 → 시스템 설계 → 상세 설계 → 구현(코딩) → 단위 테스트 → 통합 테스트 → 시스템 테스트 → 인수 테스트

설계 단계에서 이미 "이걸 나중에 어떻게 테스트할지"를 같이 정의해 둔다
</details>

<details><summary>왜 작은 시스템 개발 시 유용한가?</summary>

V 모델은 폭포수처럼 처음에 요구사항이 정확히 파악되고 안정적이라는 전제에서 잘 동작한다. 작은 시스템은 시작 시점에 요구사항을 충분히 이해할 수 있고, 단계마다 엄격하게 검증하는 비용도 감당할 만하며, 개발 도중 요구사항이 바뀔 일이 적다. 그래서 V 모델의 약점인 "변경 대응이 어렵다"는 점이 거의 드러나지 않는다. 반대로 요구사항이 계속 변하는 대형 시스템에서는 이 경직성이 그대로 부담이 된다.
</details>

### 변환 모델

> 정형 명세(formal specification)를 기반으로 하는 개발 프로세스

- 전문적인 정형 기법에 대한 지식이 필요
- 요구사항 검증을 위한 자동화된 도구가 필요

<details><summary>실제로 쓰는가?</summary>

정형 기법에 대한 깊은 수학적 지식과 전용 자동화 도구가 필요해 진입 장벽이 높기 때문에 일반 상용 소프트웨어 개발에서는 거의 쓰지 않는다. 다만 오류가 치명적인 항공우주, 원자력, 의료기기, 보안 핵심 시스템 등에서는 정확성을 증명하기 위해 제한적으로 실제 사용
</details>

## 프로세스 모델 한눈에 비교

| 모델 | 핵심 개념 | 강점 | 약점 |
|------|----------|------|------|
| 폭포수 | 순차적 단계 + 문서화 | 단순하고 관리가 쉬움 | 변경 수용이 어려움 |
| 프로토타입 | 시제품으로 요구사항 검증 | 요구사항 조기 검증, 의사소통 ⬆ | 검증 이후 변경 대응은 여전히 한계 |
| 나선형 | 위험 최소화 + 점진적 개발 | 위험 큰 대형 프로젝트에 적합 | 기간이 길고 위험 분석 비용 ⬆ |
| V 모델 | 개발 단계 ↔ 테스트 단계 대응 | 단계별 검증으로 오류 ⬇ | 변경을 다루기 어려움 |
| 변환 | 정형 명세 → 정확성 보존 변환 | 수학적으로 정확성 보장 | 전문 지식·도구 필요, 실무 사용 드묾 |

## 🔑 최종 정리 

소프트웨어 프로세스는 개발 활동을 **여러 단계로 쪼개는 것**에서 출발 <br>
이 단계를 어떻게 나누고 무엇을 먼저 다룰 것인가에 대한 서로 다른 답이 곧 **프로세스 모델**이다. <br>
가장 기본인 폭포수는 순차적이라 단순하지만 변경에 약했고, 이를 보완하려고 프로토타입과 나선형이 등장 <br> 
검증을 더 체계화해 개발과 테스트를 짝지은 것이 V 모델이고, 정확성 자체를 수학으로 보장하려는 게 정형 명세 기반의 변환 모델<br> 

> 한 줄 요약: 프로세스 모델을 고르는 일은 곧 변경과 위험을 어느 단계에서 감당할 것인가를 고르는 일이다.
