#  DevOps 변형들


## DevSecOps

> DevOps + 보안

- 과거에는 보안이 별도 보안팀에 의해 마지막 단계에서 반영됨
- 보안을 DevOps 프로세스 및 도구와 통합
- 보안 문제를 적은 비용으로 쉽고 빠르게 해결
- 보안을 개발팀과 IT 운영팀의 **공동 책임**으로 정의

### Shift Left 

> 기존에는 보안 검증이 개발 맨 끝에 있었는데 이를 개발 초반부터 끌어당겨 일찍 점검



### 보안 교육

- 애플리케이션 보안 기본 원칙
- OWASP 상위 10개 보안 리스크
- 보안 테스트 및 보안 엔지니어링 관행에 익숙해지기

### 상호 소통 문화
- 과거에는 보안이 별도 보안팀만의 일이었지만, 개발팀·운영팀·보안팀이 보안을 공동으로 책임짐

- 개발하는 사람이 코드를 짤 때부터, 운영하는 사람이 인프라를 다룰 때부터 각자 보안을 챙기면, 문제가 일찍·싸게 잡히고 책임도 분산됨

### 추적 가능성 · 감사 가능성 · 가시성

| 개념 | 의미 |
|------|------|
| 추적 가능성 | 요구사항이 어떤 형상 항목으로 구현되는지 추적 |
| 감사 가능성 | 보안 제어 준수를 보장하고 문서화 |
| 가시성 | 조직에서 수행되는 작업 상태를 확인·측정하고 알람 발생 |


## DevDataOps

> DevOps의 원리를 데이터 분석에 적용해 그 이점을 제공

> 데이터 분석 수명주기의 품질과 정보 제공의 적시성 향상이 목표

### 데이터 흐름 파이프라인

| 단계 | 하는 일 |
|------|---------|
| 생성 | 모니터링을 통해 작업 상황·결과에 대한 로그 스트림 생성 |
| 수집 | 생성된 로그를 모으고, 타임스탬프에 따라 정렬·그룹화 |
| 필터링 | 오류, 이상값, 중복 데이터 제거 |
| 스트림 처리 | 단일 로그 시스템은 실시간으로 로컬에서 처리 |
| 집적 | 통계 분석을 위해 로그를 취합해 요약 형식으로 표현 |
| 배달 | 최종 사용자·응용 프로그램에서 로그 사용 |
| 정렬 및 그룹화 | 타임스탬프에 따라 재정렬·재그룹화 |
| 저장·갱신/보관 | 효율적으로 저장·구성하고, 로그를 지속적으로 업데이트 |
| 처리 및 분석 | 데이터의 의미 있는 패턴을 발견·해석·제공 |

<details><summary>'수집'과 '정렬 및 그룹화' 단계는 무슨 차이인가?</summary>

수집은 흩어진 것을 한곳에 모으는 것 
정렬 및 그룹화는 모인 것을 제대로 줄 세우는 것


</details>

<details><summary>스트림 처리란 무엇인가? 예시가 있나?</summary>

데이터를 한꺼번에 쌓아뒀다 나중에 처리하는 게 아니라, 흘러 들어오는 즉시 실시간으로 처리하는 방식


예시:

- 카드 이상거래 탐지: 결제가 일어나는 그 순간 분석해 의심 거래를 즉시 차단
- 실시간 대시보드: 접속자 수·매출이 실시간으로 갱신되는 화면
- 로그 이상 감지·알람: 서버 에러 로그가 급증하면 즉시 경고 발송
- 추천 시스템: 사용자가 방금 클릭한 것을 반영해 바로 추천 갱신

</details>

<details><summary>실무에서는 어떤 도구로 데이터를 분석하는가?</summary>


- 수집·메시징: Kafka
- 스트림 처리: Apache Flink, Spark Streaming, Kafka Streams
- 배치 처리: Apache Spark, Hadoop 
- 워크플로 관리: Apache Airflow 
- 저장: Snowflake, BigQuery, S3 
- 로그 분석·검색: Elasticsearch, Logstash, Kibana, Splunk
- 시각화·대시보드: Grafana, Kibana, Tableau, Power BI

</details>

<details><summary>데이터 웨어하우스, 데이터 레이크는 무엇인가?</summary>

데이터 웨어하우스 : 정형화된 구조에 맞게 정제·가공한 데이터 저장
데이터 레이크 : 날것의 데이터 저장

</details>

## BizDevOps

> 개발자 + 운영 직원 + 비즈니스 팀의 협업 ➡️ 궁극적으로 수익 극대화

- 비즈니스 목표를 애플리케이션 개발·배포 단계에서도 함께 고려
- 데이터를 실시간으로 제공해 애플리케이션을 더 빠르게 변경·릴리즈

## DevOps와 기술 부채

> 기술 부채: 지금 쉬운 해결책을 선택해서 발생하는, 미래의 추가 재작업 비용

### 기술 부채의 원인

| 원인 | 설명 |
|------|------|
| 불충분한 정의 | 요구사항이 충분히 정의되지 않은 상태에서 설계 진행 |
| 높은 결합도 | 모듈화되지 않아 구성 요소가 서로 얽혀 유연하지 못함 |
| 테스트 부족 | 빠르지만 위험한 반창고식 수정으로 결함을 임시 봉합 |
| 협업 부족 | 조직 전체에 지식이 공유되지 않아 효율성 저하 |
| 지식 부족 | 유연한 아키텍처 설계를 모르면 유지보수 비용 증가 |

### 기술 부채 절감을 위한 DevOps 전략

| 전략 | 효과 |
|------|------|
| 동기 부여 | 기술 부채 해소를 별도 스프린트로 배정해  처리 |
| 자동화 활용 | 자동화로 시간을 확보해 기술 부채 완화에 투입 |
| 마이크로서비스 | 종속성을 줄여 신속한 변경과 잦은 릴리즈 가능 |
| 컨테이너화된 배포 | 수동 개입 없이 변경 코드를 컨테이너로 자동 배포 |
| API 중심 모델 | 버전별로 정의된 API로 느슨한 결합 달성 |
| 클라우드 활용 | 운영 환경 구성에 드는 노력 절감 |

<details><summary>버전별로 정의된 API가 왜 느슨한 결합을 가능하게 하는가?</summary>


어떤 서비스가 v1을 쓰고 있다고 하자. 제공자가 새 기능을 넣고 싶으면, 기존 v1은 그대로 둔 채 v2를 새로 내놓으면 된다. v1을 쓰던 쪽은 아무것도 바꿀 필요 없이 계속 돌아가고, 준비되면 v2로 옮기면 된다.

</details>

## 🔑 최종 정리 
보안을 개발 초반부터 모두의 공동 책임으로 다루면 **DevSecOps** <br>
데이터 분석 파이프라인에 적용해 적시에 양질의 정보를 뽑아내면 **DevDataOps** <br> 
비즈니스 팀까지 끌어들여 수익이라는 목표를 개발·배포에 직접 연결하면 **BizDevOps** <br>
하다 보면 **기술 부채**가 생기는데, DevOps는 이 부채를 의도적으로 관리
