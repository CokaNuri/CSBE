# DevOps 변형들

## DevSecOps

> DevOps + 보안 

- 과거에는 보안이 별도의 보안팀에 의해 마지막 단계에서 반영
- 보안을 DevOps 프로세스 및 도구와 통합
- 보안 문제를 적은 비용으로 쉽고 빠르게 해결
- 보안을 개발 팀, IT 운영 팀의 공동 책임으로 정의 

### Shift Left

> 프로세스의 오른쪽 끝에서 왼쪽으로 이동하면서 보안 이슈 처리 

- 개발은 오른쪽으로 이동, 보안은 왼쪽으로 이동 

### 보안 교육

- 애플리케이션 보안 기본 원칙
- OWASP 상위 10개 보안 리스크 
- 보안 테스트 및 보안 엔지니어링 관행에 익숙


### 상호 소통 문화

> 보안 이슈의 책임이 누구에게 있는가? 

### 추적 가능성, 감사 가능성, 가시성

- 추적 가능성 : 요구사항이 구현되는 형상 항목 추적
- 감사 가능성 : 보안 제어 준수를 보장, 문서화
- 가시성 : 조직에서 수행되는 작업 상태를 확인·측정하고 알람 발생 

## DevDataOps

> DevOps의 원리를 바탕으로, 데이터 분석에 DevOps의 이점을 제공 

> 데이터 분석 수명주기의 품질과 정보 제공의 적시성 향상이 목표 

### 데이터 흐름 파이프라인

- 생성 : 모니터링을 통해 작업 상황 및 결과에 대한 로그 스트림 생성
- 수집 : 생성된 로그 수집, 타임스탬프에 따라 정렬 및 그룹화
- 필터링 : 오류, 이상 값, 중복 데이터 제거 
- 스트림 처리 : 단일 로그 시스템은 실시간으로 로컬에서 처리 
- 집적 : 통계 분석을 위해 로그를 취합해 요약 형식으로 표현
- 배달 : 최종 사용자와 응용 프로그램에서 로그 사용
- 정렬 및 그룹화 : 타임스탬프에 따라 재정렬 및 그룹화 
- 저장, 갱신/보관 : 효율적인 저장 및 구성, 로그의 지속적 업데이트
- 처리 및 분석 : 데이터의 의미 있는 패턴에 대한 발견·해석·제공

## BizDevOps 

> 개발자, 운영 직원, 비즈니스 팀의 협업 ➡️ 궁극적으로 수익 극대화 

- 비즈니스 목표가 애플리케이션 개발 및 배포에서도 고려 
- 데이터를 실시간으로 제공하여 애플리케이션 더 빠르게 변경 및 릴리즈 

## DevOps와 기술 부채 

> 기술 부채 : 지금 쉬운 솔루션을 선택하여 발생하는 미래의 추가 재작업 비용 


### 기술 부채의 원인

- 불충분한 정의 : 요구사항이 아직 충분히 정의되지 않은 상황에서 설계 진행 
- 결합력이 높은 구성 요소 : 모듈화되지 않은 경우 유연하지 못함
- 테스트 부족 : 결함 혹은 버그를 수정하기 위해 빠르지만 위험한 반창고식 수정 진행
- 협업 부족 : 조직 전체에서 지식이 공유되지 않아 효율성 저하
- 지식 부족 : 유연한 아키텍처 설계를 모르면 유지보수 비용 증가 

### 기술 부채 절감을 위한 DevOps 전략

- 동기 부여 : 기술 부채를 개발을 위한 스프린트로 배정
- 자동화 활용 : 자동화를 통해 기술 부채를 완화하기 위한 시간 확보
- 마이크로서비스 : 종속성 X ➡️ 신속한 변경, 잦은 릴리즈 
- 컨테이너화된 배포 : 수동 개입 없이 변경된 코드를 컨테이너로 자동 배포 
- API 중심 모델 구축 : 버전별로 정의된 API ➡️ 느슨한 결합 
- 클라우드 활용 : 운영 환경 구성에 드는 노력 절감
