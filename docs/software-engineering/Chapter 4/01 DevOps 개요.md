# DevOps 개요 

> 개발 팀과 운영 팀 간에 발생하는 이슈, 애자일이 지원하지 못하는 문제를 토론하면서 개념 형성

애자일과 DevOps는 뭐가 다르지? 애자일이 해결하지 못하는 문제는 무엇이고 DevOps는 그것을 어떻게 해결하려고 하지?

## 사일로 기반 개발 방식

> 부서 간 장벽 존재·의사소통 미흡

> 자신의 태스크만 수행, 다른 부서와 정보 공유 및 협업에는 적극적이지 못함

### 문제점

- 팀 중복 작업
- 팀 간 경쟁과 협업 미흡
- 고객 니즈 반영의 어려움
- 제품 품질 저하 

### DevOps의 정의 

> 신속하게 소프트웨어를 배포하고 배포된 소트프웨어에 대한 모니터링을 통해 문제점과 개선점을 즉시 반영해 다시 배포하는 방식

> 린 사고방식, 애자일 원칙에 기반한 소프트웨어 개발 방식

## 린 사고방식과 원리

> 낭비 요소를 제거하면서 개인이 일을 통한 가치를 얻을 수 있도록 활동을 체계화하는 방법을 제공하고자 함

> 고객 만족과 직워 만족도를 일치시켜 성장을 지속

### 7가지 원칙

- 낭비 제거
- 학습 확대
- 최대한 늦게 결정
- 최대한 빨리 배포
- 팀에 권한 위임
- 통합 지원 체계
- 전체 최적화

최대한 늦게 결정하라고? 결정을 미루는 게 왜 효과적인가? 빨리 배포하라는 것과 충돌하지 않는가?

## DevOps 적용 동향

- 아마존 : 하루에 2만3천개 소프트웨어 배포
- 구글 : 하루에 5500개 소프트웨어 배포

### DevOps 도입 효과

> 산출물에 대한 품질 향상과 배포 시간 단축 

## DevOps 표준 : IEEE Std 2675

> DevOps 구현에 대한 요구사항과 가이드라인 제공 

> 4개의 프로세스 그룹에 대한 세부 프로세스 정의, 각 프로세스의 목적과 표준, 주요 산출물, 태스크 정의
