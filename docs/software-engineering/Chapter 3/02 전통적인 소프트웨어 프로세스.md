# 소프트웨어 프로세스

>서로 관련 있는 활동을 단계로 분리하여 구분한 것 

>구축, 배초, 설치, 운영 시 수행해야 하는 절차 

## 소프트웨어 프로세스의 필요성

### 블랙박스 프로세스

>하나의 통합된 개발 과정  ➡️ 요구사항을 만족시키지 못할 확률 ⬆ , 과도하게 수정하거나 재개발

### 투명한 프로세스

> 여러 단계로 분리 ➡️ 각 단계마다 사용자에게 요구사항에 대한 확인, 사용자 만족 ⬆   

> 수정 및 재개발 비용 ⬇️, 제품 출시 기간 ⬇️, 최종 품질 ⬆️

## 전통적인 프로세스 모델

### 폭포수 모델

> 순차적인 단계로 소프트웨어 개발, 각 단계마다 정해진 문서 작성

> 요구사항 분석, 설계, 개발, 테스트, 배포 

> 개발 과정에서 요구되는 변경 수용 어려움

### 프로토타입 모델

>기능 갖춘 SW가 아니라 인터페이스만 보여주거나 핵심 모듈만 우선적 개발하여 사용자에게 제공

> 첫 번째는 요구사항을 검증하기 위해 프로토타입 개발, 두 번째는 폭포수 모델에 따라 전체 시스템 개발 

- 요구사항을 프로토타입으로 빠르게 검증하여 비용과 시간 단축
- 사용자와 개발자 간 의사소통
- 빠른 오류 탐지 가능 
- 사용자 검증 과정 이후에도 변경이 발생할 수 있는데 이 점을 고려 못하는 단점은 여전히 존재 

### 나선형 모델 

> 위험을 취소화하기 위해 점진적으로 전체 시스템을 개발해나가는 모델 

> 위험의 최소화와 점진적 개발이 핵심 키워드

> 사이클 하나당 4개의 활동 
- 목표 설정 : 목표 정의, 대안 분석, 제한 사항 식별
- 위험 분석 : 예측되는 위험 사항 추출, 대처 방안 수립
- 개발 및 검증 : SW 구현, 테스트
- 평가 및 다음 계획 수립 : 산출물 평가 

장점

- 위험성이 큰 프로젝트 관리하기 적합
- 요구사항 상세히 적용 가능
- 변경되는 요구사항 반영 쉬움
- 사용자 만족도와 품질 높아짐

단점

- 프로젝트 기간이 길다
- 사이클이 많아질수록 프로젝트 관리가 어렵다
- 위험 분석에 대한 비용과 전문적 지식 필요 

### V 모델

> 폭포수 모델의 확장판, 각 단계별로 대응하는 테스트 단계가 존재 

- 단계적인 검증과 확인 과정을 거침으로써, 오류를 줄일 수 있다
- 요구사항이 정확히 이해되는 작은 시스템 개발 시 매우 유용

- 변경을 다루기 쉽지 않다 


### 변환 모델

> 정형 명세를 기반으로 하는 개발 프로세스 

- 전문적인 정형 기법에 대한 지식 요구
- 요구사항 검증을 위한 자동화된 도구 필요
