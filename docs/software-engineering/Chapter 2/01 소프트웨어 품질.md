# 01 소프트웨어 품질의 중요성

## 소프트웨어 품질을 향상한다는 것

> 사용자, 개발자, 유지보수자, 마케팅 관리자 등
> <br>**소프트웨어 개발 관련자 모두의 요구사항과 기대치를 만족**시키는 것

## 소프트웨어 품질이 중요한 이유 - 다양한 기대를 충족시켜야 한다

- 소프트웨어 품질 요소가 굉장히 많다
- 여러 관점에서 품질이 고려되어야 한다
- 다양한 품질 요소를 모두 만족시키는 것은 어렵다
- 소프트웨어의 특성을 고려한 적절한 품질 목표를 설정하는 게 중요

<br>

# 02 소프트웨어 품질 요소

| 역할       | 기대치                             | 대응 품질 요소         |
| ---------- | ---------------------------------- | ---------------------- |
| 스폰서     | 적은 비용                          | 낮은 비용, 재사용성    |
| 사용자     | 정확히 동작, 사용하기 쉬움, 고장 X | 정확성, 신뢰성, 사용성 |
| 유지보수자 | 코드 가독성, 테스트 쉬움           | 가독성, 검증 가능성    |

> 스테이크홀더마다 관심 있는 품질이 다르다.

<details>
<summary>다양한 관계자의 기대에 따라 trade-off를 고려해야 하는 실제 사례는?</summary>

- 적시성 vs 견고성 (빨리 출시 vs 버그 없이 )
- 보안성 vs 사용성 (인증 강화 vs 편리함)
- 가독성 vs 효율성 (읽기 쉬운 코드 vs 최적화)

</details>

<details>
<summary>많은 품질 중에서도 가장 중요한 품질은?</summary>

정확성이 기본 토대. 명세대로 동작조차 안 하면 다른 품질은 의미가 없다.
그 위에서 서비스 상황에 따라 우선순위를 둔다 (금융=보안·무결성,
게임=성능·사용성, 의료·항공=안전성).

</details>

<br>

## 외적 품질 요소 - 사용자가 체감하는 품질

| 요소          | 설명                                                     |
| ------------- | -------------------------------------------------------- |
| 정확성        | 명세된 요구사항대로 정확히 동작하는 능력                 |
| 신뢰성        | 주어진 조건에서 일정 기간 고장 없이 기능을 수행          |
| 견고성        | 잘못된 입력 등 **명세 밖** 비정상 상황에서도 적절히 대응 |
| 성능          | 빠른 응답·높은 처리량·적은 자원으로 효율적 처리          |
| 사용자 친숙성 | 배우고 사용하기 쉬운 정도                                |
| 가용성        | 원하는 시점에 시스템을 쓸 수 있는 비율                   |
| 보안성        | 비인가 접근·공격으로부터 시스템·데이터 보호              |
| 안전성        | 오작동이 사람·환경·재산에 해를 끼치지 않도록 보장        |
| 무결성        | 데이터·프로그램이 비인가로 변경·훼손되지 않도록 보호     |

<details>
<summary>견고성을 위해 명세에 없는 조건에도 대응하려면 어떻게 해야 할까?</summary>

- 방어적 프로그래밍(외부 입력을 신뢰하지 않는 것을 전제로 코드를 짜는 방식)
- fail-safe (고장 나도 안전한 상태로)
- fuzzing (이상한 입력을 던져 약점 찾기)

</details>

<details>
<summary>사용자 친숙성 같은 품질은 어떻게 정량화하는가?</summary>

작업 성공률, 작업 완료 시간, 오류 발생 횟수,
사용자 만족도 설문
ex) 회원가입 성공률 95%, 평균 소요시간 40초

</details>

<details>
<summary>성능은 구체적으로 어떤 지표를 보는가?</summary>

- 응답 시간: 평균 응답 속도, p95·p99
- 처리량: TPS
- 자원 사용량: CPU·메모리·네트워크·디스크 사용량
- 확장성: 부하가 늘어도 유지되는가

</details>

<details>
<summary>안정성과 신뢰성은 뭐가 다른가?</summary>

- 안정성: 고장이 발생해도 안전한가
- 신뢰성: 얼마나 자주 고장이 발생하는가
- 자주 고장 나도 안전할 수도 있고 자주 고장 안나도 그 고장이 치명적일 수 있다

</details>

<details>
<summary>보안성 - 소프트웨어에 취약점이 얼마나 있는지 어떻게 확인하나?</summary>

정적 분석, 동적 분석, fuzzing, CVE 대조 등 여러 기법으로 추정

</details>

<details>
<summary>안전성을 어떻게 정의·평가하나?</summary>

오작동이 사람·환경·재산에 위해를 끼치지 않음 고장이 나도 위험한 결과로 이어지지 않으면 안전.
지표·기법: 위험원 분석, FMEA, FTA

</details>

<details>
<summary>FMEA vs FTA</summary>

- FMEA: bottom-up. 각 요소가 고장 나면 어떤 영향을 미치는지 분석
- FTA: top-down. 사고가 일어나려면 어떤 원인들이 모여야 하는지 거꾸로 추적

</details>

<details>
<summary>무결성을 보안성과 따로 두는 이유는?</summary>

보안의 3요소 = 기밀성·무결성·가용성 (CIA). 무결성은 원래 보안의 한 축인데, 데이터가 비인가로 변경·훼손되지 않음을 강조하려고 따로 뗀 것.

</details>

<br>

## 내적 품질 요소 - 개발자가 보는 품질

| 요소        | 설명                                            |
| ----------- | ----------------------------------------------- |
| 검증 가능성 | 테스트·검증을 쉽게 할 수 있는 정도              |
| 유지보수성  | 결함 수정·기능 변경을 쉽게 할 수 있는 정도      |
| 이식성      | 다른 하드웨어·SW 환경으로 옮기기 쉬운 정도      |
| 가독성      | 코드·문서를 이해하기 쉬운 정도                  |
| 생산성      | 일정 자원으로 산출물을 만들어내는 효율          |
| 상호 운영성 | 다른 시스템과 정보를 교환하며 함께 동작         |
| 적시성      | 요구되는 시점에 맞춰 출시                       |
| 가시성      | 개발 진행 상황·상태를 외부에서 파악 가능한 정도 |

<details>
<summary>검증 가능성 - 정형 검증이란?</summary>

프로그램이 명세를 만족함을 수학적으로 증명하는 기법. 정형 검증은 모든 경우 증명. 강력하지만 비용이 커서 항공·우주·의료·원자력 등 critical 시스템에 주로 사용

</details>

<details>
<summary>유지보수성 - 유지보수 비용이 왜 신규 개발보다 큰가?</summary>

- 소프트웨어는 수명이 길다(개발은 한 번, 운영은 수년~수십 년)
- 기존 코드를 이해하는 비용이 크다 → 가독성이 중요한 이유
- 변경의 파급효과
- 보통 전체 비용의 60~80%가 유지보수. 과거 데이터·코드 규모로 예측하지만 정확히는 어렵다

</details>

</details>

<details>
<summary>가독성 - 자바 코딩 가이드라인엔 뭐가 있고 실제로 다 지키나?</summary>

Google Java Style Guide가 유명. 사람이 일일이 지키기보다 Checkstyle 같은 도구가 자동 검사(CI에 걸면 위반 시 빌드 실패로 강제됨).

</details>

<details>
<summary>재사용성이 좋은지 어떻게 판단하나?</summary>

외부 의존성이 적고, 인터페이스가 명확하며, 응집도 높고 결합도 낮을수록 재사용성이 좋다.

</details>

<details>
<summary>개발 스케줄을 어떻게 면밀히 계획하나?</summary>

작업을 잘게 쪼개 추정하고, 지연을 흡수할 버퍼(여유 시간)를 둔다. 처음부터 완벽한 계획보다 반복하며 보정하는 방식

</details>

## 프로세스 품질

> 좋은 제품은 좋은 개발 과정에서 나온다는 관점 <br>
> 제품이 아니라, 그것을 **만드는 과정**의 품질을 본다

| 요소                      | 설명                                                                 |
| ------------------------- | -------------------------------------------------------------------- |
| 프로세스 모델 적합성      | 프로젝트 특성에 맞는 개발 프로세스 모델(폭포수·애자일 등)을 골랐는가 |
| 개발 방법론 적합성        | 구조적 방법론, 객체지향 방법론 등                                    |
| 도구 적합성               | 형상관리·이슈추적·CI/CD 등 적절한 개발 도구를 갖추고 활용하는가      |
| 표준 준수성               | 코딩 규칙·문서 양식 등 정해진 표준을 따르는가                        |
| 프로젝트 데이터 관리 수준 | 프로젝트 산출물과 이력을 잘 기록·관리하는가                          |

## 응집도 (Cohesion)

> 한 모듈이 한 가지 일에 얼마나 집중하는가. **높을수록 좋다.**

| 등급        | 종류                | 설명                                     |
| ----------- | ------------------- | ---------------------------------------- |
| 최상 (강함) | 기능적 응집         | 모듈이 단 하나의 잘 정의된 기능만 수행   |
|             | 순차적 응집         | 한 요소의 출력이 다음 요소의 입력이 됨   |
|             | 교환적(통신적) 응집 | 같은 데이터를 사용하는 요소끼리 묶임     |
|             | 절차적 응집         | 정해진 순서로 실행되는 요소들을 묶음     |
|             | 시간적 응집         | 같은 시간대에 실행되어 묶임 (예: 초기화) |
|             | 논리적 응집         | 비슷한 종류의 일을 묶음 (플래그로 분기)  |
| 최악 (약함) | 우연적 응집         | 아무 관련 없는 것들이 우연히 묶임        |

## 결합도 (Coupling)

> 모듈 간 의존이 얼마나 강한가. **낮을수록 좋다.**

| 등급        | 종류        | 설명                                      |
| ----------- | ----------- | ----------------------------------------- |
| 최상 (약함) | 데이터 결합 | 필요한 데이터만 매개변수로 주고받음       |
|             | 스탬프 결합 | 자료구조 전체를 넘기고 일부만 사용        |
|             | 제어 결합   | 제어 플래그를 넘겨 다른 모듈 동작을 좌우  |
|             | 외부 결합   | 외부에서 정한 데이터 형식·프로토콜에 의존 |
|             | 공통 결합   | 전역 데이터를 여러 모듈이 공유            |
| 최악 (강함) | 내용 결합   | 다른 모듈 내부를 직접 참조·수정           |

**응집도 ↑, 결합도 ↓ 가 좋은 설계**

## 🔑 최종 정리

소프트웨어 품질은 결국 **다양한 관계자(사용자·개발자·유지보수자 등)의 기대를 만족시키는 것**이다.

- **외적 품질** : 의도한 대로 정확하고 안전하고 빠르게 잘 동작하는가?
- **내적 품질** : 검증하고 유지보수하기 쉬운가?
  - 응집도↑·결합도↓가 핵심 기준

- **프로세스 품질**: 제품 자체보다 그것을 **만드는 과정**이 좋은가?
  - 좋은 과정에서 좋은 제품이 반복적으로 나온다는 관점

> 한 줄 요약: 품질은 한 가지가 아니라 **여러 관점의 기대**이며, 서로 trade-off가 있어
> 모두 최고로 만들 순 없다. 그래서 **서비스 특성에 맞게 우선순위를 정하는 것**이 핵심이다.
