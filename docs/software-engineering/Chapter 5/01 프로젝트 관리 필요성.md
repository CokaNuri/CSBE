# DevOps 개요

> 개발 팀과 운영 팀 간에 발생하는 이슈, 그리고 애자일이 지원하지 못하는 문제를 토론하면서 형성된 개념

> 린(Lean) 사고방식과 애자일 원칙에 기반한 소프트웨어 개발 방식

<details><summary>애자일과 DevOps는 무엇이 다른가? 애자일이 해결하지 못하는 문제를 DevOps는 어떻게 해결하는가?</summary>

애자일은 개발을 어떻게 빠르게 반복하며 변화에 대응할까를 풀어냈지만 실제로 배포하고 운영하는 단계는 여전히 병목으로 남는다 <br>
개발팀과 운영팀이 분리돼 있어 벽은 여전히 존재 <br>
DevOps는 빠르게 만들고 + 배포하고 + 운영하는 것까지를 하나의 순환 고리로 묶음 
</details>

## 사일로 기반 개발 방식

> 부서 간 장벽이 존재하고 의사소통이 미흡한 방식

> 자신의 태스크만 수행하고, 다른 부서와의 정보 공유·협업에는 적극적이지 못함

### 문제점

- 팀 간 중복 작업
- 팀 간 경쟁, 협업 미흡
- 고객 니즈 반영의 어려움
- 제품 품질 저하

### DevOps의 정의

> 신속하게 소프트웨어를 배포하고, 배포된 소프트웨어에 대한 모니터링을 통해 문제점과 개선점을 즉시 반영해 다시 배포하는 방식

> 린 사고방식, 애자일 원칙에 기반함

## 린(Lean) 사고방식과 원리

> 낭비 요소를 제거하면서 개인이 일을 통해 가치를 얻을 수 있도록 활동을 체계화하는 방법을 제공

> 고객 만족과 직원 만족도를 일치시켜 성장을 지속

### 7가지 원칙

| 원칙 |  의미 |
|------|------|
| 낭비 제거  | 낭비 요소를 식별하고 삭제 |
| 학습 확대 | 개발을 효율적으로 수행할 수 있는 방법 탐색, 도메인에 대한 학습 |
| 최대한 늦게 결정  | 되돌리기 어려운 결정은 정보가 가장 많은 시점까지 미룬다 |
| 최대한 빨리 배포  | 신속하게 개발해서 사용자에게 전달 |
| 팀에 권한 위임  | 전문적인 작업은 담당자에게 위임 |
| 통합 지원 체계  | 표준화, 모듈 구성, 유연한 아키텍처 |
| 전체 최적화  | 부분이 아닌 전체 시스템 최적화 |

<details><summary>최대한 늦게 결정하라는 건 왜 효과적인가? 최대한 빠르게 배포와 충돌하지 않는가?</summary>

되돌리기 어려운 결정을 정보가 가장 많은 순간까지 미룬다는 뜻 <br>
ex) 데이터베이스 구조 변경, 아키텍처 변경
</details>

## DevOps 적용 동향

| 기업 | 배포 빈도 |
|------|-----------|
| 아마존 | 하루에 약 2만 3천 개 소프트웨어 배포 |
| 구글 | 하루에 약 5천 5백 개 소프트웨어 배포 |

### DevOps 도입 효과

> 산출물의 품질 향상과 배포 시간 단축

## DevOps 표준 : IEEE Std 2675

> DevOps 구현에 대한 요구사항과 가이드라인을 제공

> 4개의 프로세스 그룹에 대한 세부 프로세스를 정의하고, 각 프로세스의 목적·주요 산출물·활동·태스크를 정의


## 🔑 최종 정리 

> DevOps는 변경을 빠르게 받아들인다는 애자일의 정신을, 만드는 단계를 넘어 배포하고 운영하는 끝까지 늘린 흐름
