# 01 소프트웨어 품질의 중요성

## 소프트웨어 품질을 향상한다는 것

> 사용자, 개발자, 유지보수자, 마케팅 관리자 등 **소프트웨어 개발 관련자 모두의 요구사항과 기대치를 만족**시키는 것

## 소프트웨어 품질이 중요한 이유

- 소프트웨어 품질 요소가 굉장히 많다
- 여러 관점에서 품질이 고려되어야 한다
- 다양한 품질 요소를 모두 만족시키는 것은 어렵다
- 소프트웨어의 특성을 고려한 적절한 품질 목표를 설정하는 게 중요

  <br>

# 02 소프트웨어 품질 요소

| 역할       | 기대치                             | 대응 품질 요소         |
| ---------- | ---------------------------------- | ---------------------- |
| 스폰서     | 적은 비용                          | 낮은 비용, 재사용성    |
| 사용자     | 정확히 동작, 사용하기 쉬움, 고장 X | 정확성, 신뢰성, 사용성 |
| 유지보수자 | 코드 가독성, 테스트 쉬움           | 가독성, 검증 가능성    |

> 스테이크홀더마다 관심 있는 품질이 다르다.

<br>

## 외적 품질 요소 -사용자가 체감하는 품질

| 요소          | 설명                                                     |
| ------------- | -------------------------------------------------------- |
| 정확성        | 명세된 요구사항대로 정확히 동작하는 능력 (가장 기본)     |
| 신뢰성        | 주어진 조건에서 일정 기간 고장 없이 기능을 수행          |
| 견고성        | 잘못된 입력 등 **명세 밖** 비정상 상황에서도 적절히 대응 |
| 성능          | 빠른 응답·높은 처리량·적은 자원으로 효율적 처리          |
| 사용자 친숙성 | 배우고 사용하기 쉬운 정도                                |
| 가용성        | 원하는 시점에 시스템을 쓸 수 있는 비율                   |
| 보안성        | 비인가 접근·공격으로부터 시스템·데이터 보호              |
| 안전성        | 오작동이 사람·환경·재산에 해를 끼치지 않도록 보장        |
| 무결성        | 데이터·프로그램이 비인가로 변경·훼손되지 않도록 보호     |

<br>

## 내적 품질 요소 -개발자가 보는 품질

| 요소        | 설명                                            |
| ----------- | ----------------------------------------------- |
| 검증 가능성 | 테스트·검증을 쉽게 할 수 있는 정도              |
| 유지보수성  | 결함 수정·기능 변경을 쉽게 할 수 있는 정도      |
| 이식성      | 다른 하드웨어·SW 환경으로 옮기기 쉬운 정도      |
| 가독성      | 코드·문서를 이해하기 쉬운 정도                  |
| 생산성      | 일정 자원으로 산출물을 만들어내는 효율          |
| 상호 운영성 | 다른 시스템과 정보를 교환하며 함께 동작         |
| 적시성      | 요구되는 시점(납기)에 맞춰 출시                 |
| 가시성      | 개발 진행 상황·상태를 외부에서 파악 가능한 정도 |

<br>

## 프로세스 품질

- 프로세스 모델 적합성
- 개발 방법론 적합성
- 도구 적합성
- 표준 준수성
- 프로젝트 데이터 관리 수준

## ❓ 궁금한 점 & 해답

**Q. 사용자 요구사항 명세는 어떻게 모호함 없이 잘 표현할 수 있는가?**

> 유스케이스, 유저 스토리, 인수 조건을 명시한다.

ex) 사용자는 게시글 제목으로 검색할 수 있다.

검색어 "Spring" 입력 시 제목에 Spring이 포함된 게시글이 반환된다.<br>

**Q. 요구사항 명세에 정의하지 않은 조건에서도 견고하게 소프트웨어가 동작하려면 어떻게 해야 하는가? <br>모든 가능한 입력에 대해 대응하는 방법이 있는가?**

> 방어적 프로그래밍,예외 처리, 검증, fail-safe, fuzzing

**Q.사용자 친숙성 같은 품질은 어떻게 정량화하여 평가할 수 있는가?**

-> 작업 성공률,작업 완료 시간,오류 발생 횟수,사용자 만족도 설문 같은 지표를 사용한다

ex) 회원가입 성공률 : 95%
평균 소요시간 : 40초

**Q.소프트웨어에 취약점이 얼마나 존재하는지는 어떻게 확인하는가?**

-> 정적 분석,동적 분석,침투 테스트 같은 여러 기법을 사용해 추정한다. 100% 아는 건 불가능하다.

**Q.안정성을 어떻게 정의하는가? 소프트웨어가 얼마나 안전한지를 어떤 지표로 나타낼 수 있는가?**

-> 위험원 분석, FMEA, FTA, 그리고 안전 무결성 수준 등급 같은 게 있다.

**Q.무결성을 보안성과 따로 두는 이유는 무엇인가?**

-> 보안의 3요소 : 기밀성·무결성·가용성= CIA 트라이어드

무결성은 원래 보안의 한 축인데, 강조하려고 따로 뗀 것

**Q.정형 검증이란 무엇인가?**

->
프로그램이 명세를 만족함을 수학적으로 증명하는 기법

강력하지만 비용이 커서 항공,우주,의료,원자력 시스템에 주로 사용

**Q.실제 프로젝트에서 유지보수에 투입되는 시간은 어느 정도인가? 그것을 어떻게 예측할 수 있는가?**

->보통 소프트웨어 전체 비용의 60~80%가 유지보수에 들어간다고 알려져 있다

과거 프로젝트 데이터, 코드 규모등을 참고해 예측하지만 정확히 예측하기 어렵다.

**Q.재사용성이 좋은지 어떻게 판단하는가?**

->외부 의존성 적음,인터페이스 명확, 응집도·결합도

**Q.이식성이 얼마나 중요한가? 컨테이너 기술로 다 해결 가능한가?**

-> 도메인마다 중요도가 다르다.

컨테이너는 실행 환경을 통일한다.
하지만 CPU 아키텍처 차이, 특수 하드웨어 의존등은 해결할 수 없다. 따라서 이식성 문제를 줄여줄 뿐 제거하진 못한다.

**Q. 자바 코딩 가이드라인에는 어떤 것이 있는가? 실제로 이 가이드라인을 다 따르면서 코딩하는가?**

->Google 의 Java Style Guide이 유명하다.
Checkstyle같은 도구가 자동 검사한다.

**Q.개발 스케줄을 어떻게 면밀하게 계획할 수 있는가?**

-> 작업을 잘게 쪼개고 버퍼를 둔다. 완벽한 계획보다는 반복하면서 보정하는 방식을 사용한다.

**Q.개발을 하면서 문서화해야 할 정보에는 어떤 것들이 있는가?**

->요구사항 명세서, 시스템 구성도, API 명세서, DB 스키마,운영 가이드 등

**Q.다양한 관계자의 기대에 따라 trade-off를 고려해야 하는 실제 사례는 무엇이 있는가?**

-> 적시성 vs 정확성
보안성 vs 사용성
가독성 vs 효율성

**Q.많은 품질 중에서도 가장 중요한 품질은 무엇인가?**

-> 정확성을 기본 토대로 서비스 상황에 따라 우선순위를 둔다

## 2차 질문

**Q.유스 케이스와 유저 스토리는 뭐가 다른가?**

-> 유저 스토리는 애자일에서 많이 쓰고 짧고 비형식적이다. 유스 케이스는 더 길고 구조적으로 사용자와 시스템의 상호작용을 자세히 기술한다.

**Q.유지보수 비용이 왜 신규 개발 비용보다 큰가?**

-> 소프트웨어는 수명이 길다. 기존 코드를 이해하는 비용이 크다. 가독성, 응집도·결합도 등의 개념과 연결되어 있다.

**Q. FMEA, FTA는 무엇인가?**

-> FMEA는 bottom-up 방식으로 각 요소가 고장 나면 어떤 영향을 미치는지 분석하는 방법이다.

FTA는 top-down 방식으로 사고가 발생하려면 어떤 원인들이 모여야 하는지 거꾸로 추적하는 방식이다.

**Q.방어적 프로그래밍이란 무엇인가?**

-> 입력과 외부 환경을 신뢰하지 않는것을 전제로 코드를 짜는 방식이다. 들어오는 값을 검증하고 예외 처리를 한다.

**Q.fail-safe는 무엇인가?**

-> fail-safe는 시스템이 고장이 나더라도 안전한 상태로 빠지게 설계하는 것.

ex) 은행 송금 오류 - 둘 다 롤백

**Q.fuzzing은 무엇인가?**

-> 의도적으로 이상한 입력값을 넣어보는 테스트 방법

**Q.응집도와 결합도를 어떻게 판단하는가?**

-> 둘 다 등급으로 판단한다.

## 응집도 (Cohesion)

> 한 모듈이 한 가지 일에 얼마나 집중하는가. **높을수록 좋다.**

| 등급        | 종류                | 설명                                     |
| ----------- | ------------------- | ---------------------------------------- |
| 최상 (강함) | 기능적 응집         | 모듈이 단 하나의 잘 정의된 기능만 수행   |
|             | 순차적 응집         | 한 요소의 출력이 다음 요소의 입력이 됨   |
|             | 교환적(통신적) 응집 | 같은 데이터를 사용하는 요소끼리 묶임     |
|             | 절차적 응집         | 정해진 순서로 실행되는 요소들을 묶음     |
|             | 시간적 응집         | 같은 시간대에 실행되어 묶임 (예: 초기화) |
|             | 논리적 응집         | 비슷한 종류의 일을 묶음 (플래그로 분기)  |
| 최악 (약함) | 우연적 응집         | 아무 관련 없는 것들이 우연히 묶임        |

## 결합도 (Coupling)

> 모듈 간 의존이 얼마나 강한가. **낮을수록 좋다.**

| 등급        | 종류        | 설명                                      |
| ----------- | ----------- | ----------------------------------------- |
| 최상 (약함) | 데이터 결합 | 필요한 데이터만 매개변수로 주고받음       |
|             | 스탬프 결합 | 자료구조 전체를 넘기고 일부만 사용        |
|             | 제어 결합   | 제어 플래그를 넘겨 다른 모듈 동작을 좌우  |
|             | 외부 결합   | 외부에서 정한 데이터 형식·프로토콜에 의존 |
|             | 공통 결합   | 전역 데이터를 여러 모듈이 공유            |
| 최악 (강함) | 내용 결합   | 다른 모듈 내부를 직접 참조·수정           |

**응집도 ↑, 결합도 ↓ 가 좋은 설계**
