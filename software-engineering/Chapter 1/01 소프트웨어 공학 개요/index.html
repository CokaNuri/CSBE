    소프트웨어 고장 사례¶

    핵심 : 소프트웨어 결함은 돈·생명까지 앗아간다
    → 그래서 소프트웨어 공학이 필요하다. 각 사례를 "어떤 품질이 무너졌나"로 읽어보자

    화성 기후 관측 위성 (1999)¶

    • 원인: 단위 불일치. 국제 표준 단위인 N·s로 표시해야 하는데 미국식 lbf·s로 표시
    • 결과: MCO 위성이 사라짐 → 큰 경제적 손실
    • 교훈: 표준을 잘 따르자. 문서화를 잘 하자.

    자동차 해킹¶

    • 원격으로 차량 제어 시스템에 침입해 핸들·브레이크·엔진 조작 가능 (예: 2015 짚 체로키)
    • 교훈: 보안성이 곧 안전성과 직결. 차가 인터넷에 연결되며 보안 결함이 인명 위협으로

    보잉 737 MAX8 추락¶

    • 원인: MCAS라는 SW 기반 제어 시스템에 입력되는 받음각(AOA) 센서 값 오류
      하나의 센서 값만 이용하여 동작하도록 개발
    • 결과: 두 사고로 346명 사망, 전 세계 737 MAX 약 20개월 운항 정지
    • 교훈: 단일 장애점. 최소 두 센서를 교차 확인했어야 함. 견고성·안전성 설계 실패

    테슬라 자율주행¶

    • 오토파일럿/FSD 사고. 운전자 과신 + AI가 특정 상황 오인
    • 결과: 테슬라 모델 S 오토파일럿 사용 중 운전자 사망 사고
    • 교훈: AI 견고성, 보조 시스템을 최종 결정에 쓰면 안 된다는 범위 준수

    기타 소프트웨어 사고 일지¶

    • Therac-25(방사선 치료기 SW 결함 → 환자 사망)
    • Ariane 5(숫자 변환 오버플로로 로켓 폭발)
    • 산업·과학기술 분야에서 SW 결함으로 인한 사고 발생
    Q. 각 사고 사례를 막을 수 있는 소프트웨어 공학 기법은 무엇일까? - 화성 위성 → 인터페이스 명세 + 검증·확인 - 자동차 해킹 → 보안 설계, 침투 테스트 - 보잉 737 → 다중화(센서 2개 이상 교차 확인), 견고성·안전성 설계 - 테슬라 → 범위 준수(보조 시스템임을 명확히), 운전자 개입 보장 설계
    Q. 자동차·우주·비행기 등의 분야 말고 일반적인 생활 편의 분야에서도 심각한 사고가 발생한 사례가 있나? - 크라우드스트라이크 사태(2024): 보안 SW의 잘못된 업데이트로 전 세계 윈도우 850만 대 마비. 항공·은행·병원 중단, 포춘 500대 기업에만 약 54억 달러 손실. 역대 최대 IT 장애. - AWS 대규모 장애(2025): 클라우드 백엔드 오류 하나가 번져 런던증권거래소·코인베이스·국세청까지 멈춤. 시간당 약 7500만 달러 손실 추정. - 교훈: 단일 장애점 + 점진적 배포(롤아웃)의 부재 + 테스트 미흡. 가용성도 핵심 품질

    소프트웨어 위기¶

    원인과 증상¶

    • 원인: 복잡성 증가 + 소프트웨어 공학의 상대적 미성숙
    • 증상: 예산·일정 초과, 낮은 품질, 유지보수 곤란
    Q. 소프트웨어 공학이 성숙해지고 정립된 과정이 어떻게 될까? - 1968년 NATO 소프트웨어 공학 학회에서 소프트웨어 위기라는 말과 소프트웨어 공학이라는 용어가 공식 등장 → 출발점 - 1970년대: 구조적 프로그래밍·구조적 방법론 정립 - 1980~90년대: 객체지향, CASE 도구, 프로세스 모델(폭포수·나선형) 발전 - 2000년대: 애자일 선언(2001), 이후 DevOps로 이어짐

    소프트웨어 개발이 어려운 이유¶

    SW 개발은 창의적인 활동 → 엔지니어의 지식과 경험에 따라 결과물 천차만별

    • 의사소통 문제: 다양한 역할 담당자 → 소통 오류 발생
    • 순차 특성: 복잡한 동작 구조를 선형적인 코드/문서로 표현하는 어려움
    • 프로젝트 복잡성: 개발 기간, 기술 종류, 개발자 수, 목적 등이 다양
    3차원적 상황을 2차원 평면에 정리한다는 게 무슨 말일까? 소프트웨어의 실제 동작은 여러 요소가 동시에·입체적으로 얽혀 일어나는데, 이를 표현하는 코드·문서는 위에서 아래로 읽는 선형 구조다. 즉 입체적인 동작을 평면에 눌러 담아야 해 본래 구조가 잘 안 드러나고 이해가 어렵다는 의미

    소프트웨어 특성¶

    왜 소프트웨어를 단순한 개발 활동으로 간주할 수 없는가?

    왜 체계적인 공학적 기술을 도입해야 하는가?

    • 개발 비용: 분석·설계, 테스트가 코드 개발보다 더 중요
    • 분석·설계(40%), 테스트(40%), 코드 개발(20%)
    • 유지보수: 운영·유지 비용이 개발 비용의 2배
    • 고장 그래프: SW는 일정 시간 후 변경 과정을 거친다
    • 수정이 반복되면 코드가 더 복잡해지고 이해하기 어려워져 오류 발생 ⬆️

    인공지능 시대의 소프트웨어 위기¶

    • 너무 많은 학습 알고리즘이 무분별하게 응용 SW 개발에 적용되는 문제
    • 자율주행차, 스마트팩토리 로봇 제어 등 안전 중심 SW에도 적용 ➡️ 인명 피해 위험
    • 훈련된 데이터를 조금만 변형해도 다른 데이터로 인식하는 문제 (적대적 입력)

    소프트웨어 공학 기술의 적용¶

    위기의 답으로 등장한 것이 소프트웨어 공학

    소프트웨어 공학적 기법의 종류¶

    소프트웨어 품질을 높이고, 신속·정확하게 SW를 전달하는 게 목표

    • 구조적 프로그래밍: 순차·선택·반복 세 구조로 프로그램 구성
    • 객체지향 프로그래밍: 직관적 이해, 유지보수성·재사용성 ⬆️
    • 소프트웨어 컴포넌트 및 재사용: 검증된 부품을 조립해 개발 시간 단축, 대규모 SW 개발
    • 개발 프로세스: 폭포수·애자일 등 체계적 SW 개발 절차
    • 검사 및 검증: 테스트 등으로 SW가 제대로 동작하는지 확인
    • 형상 관리: 버전·변경 이력 통제
    • 소프트웨어 아키텍처: 변경에 강한 SW를 위해 상호 독립성이 높은 구성 요소를 식별하고 관계를 정의
    객체지향이 구조적 프로그래밍보다 무조건 좋은 것일까? 구조적 프로그래밍의 장점은? 무조건 좋은 건 아니다. 구조적 프로그래밍의 장점: 단순하고 직관적이며, 작은 프로그램·순차적 처리(예: 데이터 변환, 임베디드, 알고리즘)에서는 더 빠르고 효율적. 객체지향은 규모가 크고 변경·재사용이 많은 시스템에서 빛나지만, 추상화 계층이 많아 작은 작업엔 오히려 과한 복잡성이 될 수 있다.
    나선형 방식, V 모델, DevOps 개발 방식이 무엇일까? - 나선형(Spiral): 반복할 때마다 위험 분석을 핵심에 두는 모델. 위험 큰 대형 프로젝트에 적합 - V 모델: 폭포수를 V자로 변형 각 개발 단계에 대응하는 테스트 단계를 짝지음(요구분석↔인수테스트, 설계↔통합테스트) 검증을 강조 - DevOps: 개발(Dev)과 운영(Ops)을 통합. 빌드·테스트·배포를 자동화(CI/CD)해 빠르고 지속적으로 출시·운영. 애자일의 연장선
    어떤 SW 아키텍처 종류가 있을까? 많이 사용되는 아키텍처는? 대표적으로 계층형(Layered), MVC, 클라이언트-서버, 마이크로서비스, 이벤트 기반등. 웹 서비스에선 계층형·MVC가 기본으로 매우 많이 쓰이고, 큰 서비스는 마이크로서비스로 확장한다.

    소프트웨어 공학의 정의와 원리¶

    정의¶

    • SW 위기에서 벗어나기 위한 공학적 기법
    • 전체 수명주기 지원
    • 사용자 요구사항 충족
    • 좋은 품질의 소프트웨어 개발

    원리¶

    원리 설명
    엄격성·정형성 지나친 자유로움을 제한해 정확하게 기술
    관심사 분리 복잡한 문제를 분리하여 별도로 관리
    모듈화 독립된 작은 단위로 만들어 쉽게 분리·조합 👌
    추상화 핵심만 남기고 세부사항은 감춤 ➡️ 관심사 잘 표현
    변경 예측 미래의 변경을 미리 고려해 설계
    일반화 더 넓게 재사용 가능한 형태로
    점진성 조금씩 단계적으로 발전시킴
    명세화 무엇을 만들지 명확히 정의·문서화
    엄격성과 정형성은 뭐가 다른가? - 엄격성(rigor): 모호함 없이 정밀하게 다룬다는 태도·수준 - 정형성(formality): 그 엄격함의 최고 단계로, 수학적 표기·형식 언어로 기술하는 것
    모듈화의 단위는 무엇인가? 함수 → 클래스 → 패키지 → 컴포넌트 → 서비스. 고정된 한 가지가 아니라 보는 수준에 따라 다름
    일반화의 원리는 어떻게 적용할 수 있는가? 특정 값·타입·대상을 더 넓게 끌어올리는 것: 매개변수화, 인터페이스, 제네릭

    🔑 최종 정리¶

    소프트웨어를 제대로 안 만들면 재산·인명 사고가 발생한다. → 그래서 좋은 품질의 소프트웨어를 만들어야 한다. → 그런데 SW의 어떤 특성 때문에 좋은 소프트웨어 개발이 어려운가?

    ① 소프트웨어는 복잡하고 전체 수명을 다 관리해야 한다

    • 단순 코드 개발보다 설계·테스트·유지보수에 더 많은 비용이 든다
    • → 해법 원리: 관심사 분리, 모듈화, 추상화,변경 예측

    ② 다양한 사람들의 요구사항을 모두 충족시켜야 한다

    • 관계자가 많아 의사소통 비용이 크다
    • → 해법 원리: 명세화, 문서화, 표준화

    한 줄 요약: 소프트웨어는 복잡하고 수명이 길며 사람이 많이 얽혀 만들기 어렵다. 그래서 모듈화·추상화·관심사 분리·변경 예측으로 복잡성과 변경에 대응하고, 명세화·문서화·표준화로 소통 문제를 줄이는 것이 소프트웨어 공학이다.
