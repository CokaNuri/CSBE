    03 애자일 및 XP 프로세스¶

    애자일 방법론¶

    애자일 개발을 가능하게 해주는 방법론 전체

    애자일 선언문의 네 가지 가치

    더 중시하는 것 ~보다
    개인과 상호작용 공정과 도구
    작동하는 소프트웨어 포괄적인 문서
    고객과의 협력 계약 협상
    변화에 대응하기 계획을 따르기

    신속한 소프트웨어 개발¶

    잦은 요구사항 변경을 어떻게 해결하는가가 핵심

    • 사용자의 검토 과정을 반복적으로 진행하는 것이 필요
    • 가시적인 소프트웨어를 보여주는 개발 접근 방법

    애자일 : XP 프로세스¶

    핵심 키워드: 반복 개발 + 철저한 테스트

    실천 사항¶

    실천 사항 의미
    증분 계획 작은 단위로 계획을 세우고 점진적으로 확장
    작은 배포 작동하는 소프트웨어를 자주, 조금씩 배포
    단순 설계 지금 필요한 만큼만 단순하게 설계
    테스트 우선 개발 코드를 짜기 전에 테스트부터 작성
    리팩토링 동작은 유지한 채 내부 구조를 지속적으로 개선
    페어 프로그래밍 두 명이 함께 코딩
    공동 소유권 누구나 어떤 코드든 수정 가능
    지속적 통합 변경을 자주 통합하고 즉시 검증
    지속 가능한 개발 속도 일정한 페이스 유지
    현장 고객 고객이 개발팀과 함께 상주하며 즉시 피드백

    엔지니어링 사이클¶

    코딩 → 테스트 → 사용자 피드백 → 설계 → 코딩 → …

    배포 사이클¶

    사용자 스토리 선정 → 스토리 분할 및 태스크 식별 → 배포 계획 수립 → 개발/통합/테스트 → SW 배포 → 시스템 평가 및 피드백 → …

    애자일 : 스크럼 프로세스¶

    반복적인 개발 관리에 초점, 피드백 중심의 경험적 접근 방법

    흐름: 제품 백로그에서 한 스프린트에 완료할 수 있는 작업 선택 → 스프린트 백로그 구성 → 스프린트 진행(개발) → 결과 점검·개선.

    구성 요소 설명
    제품 백로그 제품에 필요한 모든 요구사항의 우선순위 목록
    스프린트 백로그 이번 스프린트에 완료하기로 한 작업 목록
    번다운 / 번업 차트 남은 일과 완료한 일을 시각화
    데일리 스크럼 매일 짧게 진행 상황을 공유하는 미팅
    스프린트 리뷰 스프린트 결과물을 점검하는 미팅
    스프린트 회고 일하는 방식 자체를 돌아보는 미팅
    XP와 스크럼의 차이는? XP는 엔지니어링쪽에 더 중점을 두고, 스크럼 프로세스는 프로젝트와 팀 관리에 더 중점을 둔다
    데일리 스크럼은 왜 하는가? 매일 짧게 모여 어제 한 일, 오늘 할 일, 막힌 점 세 가지를 공유하기 위해서 문제를 빠르게 노출하고 조율하는 게 핵심
    스프린트 리뷰와 회고 미팅은 무엇이 다른가? 리뷰는 만든 결과물을 보고, 회고는 일하는 방식을 본다

    카오스 엔지니어링¶

    기존 서비스에 임의의 결함을 의도적으로 주입해, 시스템이 실제 장애 상황에 잘 대응하는지 검증하는 기법

    • 장애가 터지기 전에 약점을 미리 찾아내 복원력을 높이는 것이 목적
    • 운영 중인 시스템의 안정성을 능동적으로 시험

    DevOps¶

    개발(Dev)과 운영(Ops), 사용자 피드백을 하나로 상호 연결하는 문화·방식

    요구사항을 지속적으로 반영하고, 높은 가용성을 제공

    핵심 요소 내용
    자동화 빌드·테스트·배포 과정을 자동화
    반복 유지보수라는 기존 개념 대신, 운영 중 모니터링된 요구사항을 지속적으로 반영
    지속적 개선 피드백을 바탕으로 계속 다듬어 나감
    CI / CT / CD 지속적 통합, 지속적 테스트, 지속적 배포
    릴리즈와 배포는 무슨 차이인가? 배포는 새 버전의 소프트웨어를 서버 등 운영 환경에 설치·반영하는 기술적 행위 릴리즈는 그 기능을 실제 사용자에게 노출해 사용할 수 있게 하는 행위 사용자에게 새 버전이 닿느냐 안 닿느냐 차이

    🔑 최종 정리¶

    애자일은 변경은 당연한 것이라 전제하고 잦은 요구사항 변경을 반복적 협업으로 흡수하려는 사고방식

    한 줄 요약: 애자일은 변경을 전제로 삼아, 짧은 반복과 피드백의 주기를 개발에서 운영까지 확장한 흐름이다
