    DevOps 프로세스와 프랙티스¶

    DevOps 성공 전략¶

    사일로 개발 방식을 타파하고 부서 간 효과적인 의사소통이 필요

    성공적인 DevOps 구현을 위한 권장 사항¶

    • DevOps 필요성 평가
    • 기존 사일로 방식 타파
    • 고객 중심의 만족도 향상
    • 작은 시작과 점진적 확장
    • 가능한 모든 곳의 자동화
    • 호환성을 보장하는 도구 사용
    • 개인 성과와 팀 성과를 함께 검토
    • 프로젝트 정보의 실시간 가시성 확보
    • 지속적 통합 및 배포(CI/CD)
    • 모니터링 및 피드백을 통한 개선

    DevOps 프로세스¶

    애플리케이션 배포·운영 환경과 개발·테스트 환경 간의 격차를 해소하는 것이 목적

    단계 주요 활동
    계획 요구사항 명세, 배포 척도 조정, 배포 계획·시기·비즈니스 사례 정의
    코딩 소프트웨어 설계, 제품 형상 설계·정의, 소스 코드 작성, 단위 테스트
    빌드 통합 테스트, 제품 형상 구성, 배포용 애플리케이션 준비, 버전 관리
    테스트 요구사항 만족도·회귀·성능 테스트, 보안 및 취약성 분석, 형상 테스트
    릴리즈 형상에 따라 패키징, 애플리케이션 배포
    설치 사용자 환경에 설치, 인프라(스토리지·DB·네트워크) 구축, 사용자에게 오픈
    운영 및 모니터링 인프라·사용자 경험 모니터링, 개발 척도 달성 여부 확인, 결과를 개발팀에 피드백
    '제품 형상 설계', '형상 구성', '형상 테스트'가 각각 무엇인가? 제품 형상 설계·정의는 이 제품이 어떤 구성 요소들로 이뤄지는가를 정함 어떤 모듈·라이브러리·의존성·버전이 제품을 구성하는지 관리 대상을 식별·정함 형상 구성은 그렇게 정의한 요소들을 실제로 하나의 빌드로 조립 특정 버전의 부품들을 모아 일관된 하나의 결과물로 묶음 형상 테스트는 그렇게 조립된 구성이 통째로 올바른지 검증 의도한 버전의 부품들이 빠짐없이 들어갔고, 서로 잘 맞물려 동작하는지 확인
    패키징은 정확히 무엇인가? 빌드된 애플리케이션을, 그것이 실행되는 데 필요한 모든 것(라이브러리·의존성·설정·메타데이터)과 함께 하나의 배포 가능한 단위로 묶는 작업
    이렇게 묶어두면 어느 환경으로 옮겨도 동일하게 설치·실행할 수 있음 ex) 자바의 JAR/WAR 파일, 도커 이미지

    DevOps와 클라우드¶

    클라우드를 통해 DevOps를 구현하면 서비스 개발·배포를 편리하게 할 수 있음

    • On-Demand 시점의 가용성·접근성을 제공받을 수 있음
    • 다양한 테스트가 쉽게 이루어질 수 있음

    풀 스택 배포¶

    클라우드 환경에 애플리케이션 서비스 배포와 실행 환경 구성에 대한 배포를 함께 진행

    • 분리 배포: 클라우드 환경 프로비저닝을 애플리케이션 서비스 배포와 분리해 배포
    클라우드 환경 프로비저닝은 무엇이고, 실행 환경 구성은 무엇인가? 프로비저닝은 애플리케이션이 돌아가는 데 필요한 인프라 자원을 마련·할당하는 일
    클라우드에서 서버, 스토리지, 네트워크, 데이터베이스 같은 자원을 새로 띄우고 배정하는 것 실행 환경 구성은 그 위에서 앱이 실제로 실행될 수 있도록 환경을 갖추는 일
    운영체제, 런타임(JVM, Node.js), 미들웨어, 각종 설정을 올려 앱이 동작할 토대를 만듦

    클라우드 서비스 모델¶

    모델 제공 범위 사용자가 신경 쓸 것
    IaaS 네트워크·저장소·서버 등 가상화된 인프라만 제공 인프라 위의 모든 것 (개발과 인프라 관리 간 사일로 존재)
    PaaS 인프라 + 실행 플랫폼까지 제공 애플리케이션과 데이터의 생성·관리만 고민하면 됨
    SaaS 완성된 소프트웨어를 서비스로 제공 사용만 하면 됨

    DevOps 측정 척도¶

    신속하고 지속적으로 배포하는 것이 목적이므로, 그 성과를 수치로 측정

    측정 척도 의미
    설치 주기 / 규모 / 시간 배포를 얼마나 자주·얼마만큼·얼마나 빨리 하는가
    소요 시간 작업을 시작해 배포되기까지 걸리는 시간
    고객 티켓 사용자가 제기한 문제 건수
    자동 테스트 성공률 자동화된 테스트의 통과 비율
    결함 제거율 발견된 결함을 제거한 비율
    가용성 서비스가 정상 동작하는 시간 비율
    서비스 수준 협약(SLA) 약속한 서비스 품질 기준
    서비스 활용·트래픽 실제 사용량과 부하
    서비스 성능 응답 속도 등 성능 지표
    결함 발견 간 평균 시간(MTBF) 결함과 결함 사이의 평균 간격
    결함 복구 간 평균 시간(MTTR) 장애 발생 후 복구까지 걸리는 평균 시간
    이렇게 많은 척도 중 가장 중요한 것은 무엇일까? - 설치 주기: 얼마나 자주 배포하는가 - 소요 시간: 코드 작성부터 배포까지 얼마나 걸리는가 - 변경 실패율: 배포가 장애로 이어지는 비율 - 평균 복구 시간: 장애가 나면 얼마나 빨리 복구하는가

    🔑 최종 정리¶

    한 줄 요약: DevOps의 실행은 사일로를 허문 자동화된 순환 프로세스를 클라우드 위에서 돌리고, 속도와 안정성의 균형을 지표로 확인하는 일이다.
